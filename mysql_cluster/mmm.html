# MMM

# 功能：

  • MMM监控Mysql主从复制的健康情况，链路是否正常，主从是否存在延迟。

  • 当出现主从延迟时，或者主从复制链路中断时，会对主服务进行故障转移，并且将虚拟IP迁移到新的主上。自动配置其他从对新主的复制。

  • 提供了主，写虚拟IP，在主从服务器出现问题时，可以自动迁移虚拟IP。

    • 对每一个服务器，都会提供一个虚拟IP，其中包括一个写的虚IP和多个读的虚IP。

    • 写的虚IP只能在两个主服务器之间，进行切换

    • 读的虚IP则可以在集群所有机器上进行切换

# 优缺点

# 优

  • 使用Perl脚本语言开发及完全开源

  • 提供来读写VIP（虚拟IP），使服务器角色的变更对前端应用透明

    • 解决问题：当主数据库出现故障变更IP后，如何通知应用对新的主DB重新建立链接。

  • MMM提供了从服务器的延迟监控

  • 提供了主数据库故障转移后，从服务器对新主重新同步功能

# 缺点

  • 发布时间比较早，不支持MySQL新的功能

    • 比如MySQL 5.6的GTID复制、多线程复制，都不支持

    • 最新版本距离现在已经有好几年的时间，存在一些Bug，需要DBA手工修复。

  • 没有读负载均衡的功能

    • 需要单配使用Lvs、F5等配合使用，才可以实现负载均衡，增加管理负担

  • 在进行主从切换时，容易造成数据丢失

    • MMM对这快的处理并不安全，只是简单粗暴的找到新的主库的日志点，然后使所有的从库对这个日志点来进行同步。在繁忙的系统中，可能会造成数据丢失的情况。

    • 主备的数据，不一定是最新，存在一步复制的数据差异性。 强制让从库复制主备的数据，导致数据丢失

    • 主备的数据不一定比从服务器的数据新。

  • 监控服务存在单点故障

# Faq

  • 出现故障自动故障转移，并且配置其他从对新主的复制时，要解决的问题

    • 如何找到从库对应的新的主库日志点的日志同步点
    • 如果存在多个从库出现数据不一致的情况如何处理

    结论

    MMM对这快的处理并不安全，只是简单粗暴的找到新的主库的日志点，然后使所有的从库对这个日志点来进行同步。在繁忙的系统中，可能会造成数据丢失的情况。

  • 出现故障时，3M自动转移故障的操作步骤

    • 当监控服务器发现主服务无法访问时，MMM监控服务 就会把写VIP切换到主备服务器。
    • 同时设置原来的主为 read-only 模式

  • 从服务器出现故障时，mmm会把有故障的从剔除吗？重点

# 工作模式

  • 拓扑图

  • 主动被动模式的主主复制

    • 只有一个主，对外提供服务
    • 另外一个主处于备用状态，只能用于查询，并且设置为Read-only模式

# 部署资源

资源名称 数量 说明
主DB服务器 2 用于主备模式的主主复制配置
从DB服务器 0-N 可以配置0台或多台从服务器
监控服务器 1 用于监控MySQL复制集群
IP地址 2 * (n-1) n为MySQL服务器的数量,每台一个物理IP，一个虚拟IP
监控用户 1 用于监控数据库状态的MySQL用户 (replication client)
代理用户 1 用于MMM代理的MySQL用户 (super、replication，process)
复制用户 1 用户配置MySQL复制的用户 (replication slave)

# 开始部署

部署MMM之前，必须先配置好，主主复制，外加一个从。

日志点主从复制

# 配置yum

wget https://dl.fedoraproject.org/pub/epel/epel-release-latest-7.noarch.rpm
rpm -ivh epel-release-latest-7.noarch.rpm --force --nodeps
1
2

# 安装MMM代理服务

  • 每台集群中的机器都要安装。
yum search mmm*
yum install mysql-mmm-agent.noarch
1
2

# 安装MMM监控服务

  • 只需在监控服务器上安装
yum install mysql-mmm*
1

# 创建监控服务账号

  • 在主服务器上创建，会同步到其他从服务
grant replication client on *.* to 'mmm_monitor'@'172.17.183.%' identified by 'demo@123456';
1

# 创建代理服务账号

  • 在主服务器上创建，会同步到其他从服务
  • 这个账号主要用于故障转移，主从切换。
    • 改变 read-only的模式
    • 改变 从服务器对应的主。
grant SUPER,REPLICATION CLIENT,PROCESS on *.* to 'mmm_agent'@'172.17.183.%' identified by 'demo@123456';
FLUSH PRIVILEGES;
1
2

# 配置代理服务参数

  • 下面的配置参数在所有代理服务器上必须保持一致
vim /etc/mysql-mmm/mmm_common.conf
1
active_master_role  writer //主服务只能用来写

<host default>
    cluster_interface       eth0
    pid_path                /run/mysql-mmm-agent.pid
    bin_path                /usr/libexec/mysql-mmm/
    replication_user        slave           //复制账号
    replication_password    demo@123456     //复制账号密码
    agent_user              mmm_agent       //代理账号
    agent_password          demo@123456     //代理账号密码
</host>

<host db1>
    ip      172.17.183.121
    mode    master
    peer    db2
</host>

<host db2>
    ip      172.17.183.122
    mode    master
    peer    db1
</host>

<host db3>
    ip      172.17.183.123
    mode    slave
</host>

//写服务，db1和db2，为两个主。IP为虚拟
<role writer>
    hosts   db1, db2
    ips     172.17.183.140
    mode    exclusive
</role>

//读服务，db1, db2, db3都可以读
<role reader>
    hosts   db1, db2, db3
    ips     172.17.183.141, 172.17.183.142, 172.17.183.143
    mode    balanced
</role>

# 启动服务

启动服务顺序：先启动代理，后启动监控

  • 启动代理
/ect/init.d/mysql-mmm-agent start
1
  • 启动监控
/ect/init.d/mysql-mmm-monitor start
1

注意

  • 启动成功后，查看一下虚拟IP是否配置上。在每一台机器上都可以看一下

    ip addr
    1

    • 启动
    /etc/init.d/mmm_control start
    1
    • 查看集群状态
    mmm_control show
